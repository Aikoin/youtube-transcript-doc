# 清洗 + 翻译规范

自动字幕是"毛坯"：满是语气词、口吃重复、ASR 误识。目标是产出**读起来像书面语、但仍是说话人原话**的逐字稿——清洗，不是改写，更不是摘要。

## 清洗（每一段都做）
1. **删语气词**：um、uh、作填充的 like / you know / kind of / sort of。
2. **删口吃重复**：`it it's`→`it's`、`the the`→`the`、`many many`→`many`、`should should`→`should`。
3. **补标点大小写**：让每段成为通顺的书面句子。
4. **纯语气词段整段删除**：如单独成段的 "Um,"、"Yeah."、"Y"（`parse_srt.py` 已自动删，人工复核）。
5. 保留说话人的实际用词与含义，**不要意译、不要概括**。

## 修正明显的 ASR 错误
人名、专有名词、术语最常被识别错。建立一张"术语修正表"贯穿全文替换。常见类型：
- 人名：说话人和被提及者的名字（结合视频标题/简介确认正确拼写）。
- 公司/产品名：大小写规范化（如 OpenAI、GitHub、Notion、Anthropic）。
- 技术术语：如 `Lauras`→`LoRA`、`chat GBT`→`ChatGPT`、连字符统一（`on policy`→`on-policy`）。
- 谐音误识：靠上下文判断（如某心理学家名字被拆错）。

**做法**：先通读一遍，列出该视频专属的修正表，再交给翻译子任务统一应用。

## 翻译
- 忠实、通顺的简体中文，对照清洗后的英文逐段翻译。
- 术语保留业界通用译法或原文（如 LoRA、KV cache、RAG 保留英文）。

## 分段并行
逐字稿常有近百段。可拆成 2–3 批，用并行子 agent 各译一批，**共享同一张术语修正表**保证一致，
每批输出 `{index: {segs:[{speaker,zh,en_keep}], key}}` 的 JSON，最后合并。要求子 agent 用 `json.dump` 落盘以保证 JSON 合法。

## 说话人归属 + 重点标记（在清洗/翻译时一并完成）
除了 zh/en_keep，每段还要判定这些，写进 `content.json` 的 `paras`：

1. **多说话人拆分（segs）**：ASR 常把「提问 + 回答」这类多人一来一回并进同一段。**必须**把这种合并段按语义切点拆成有序的 `segs` 列表，每个 seg 是一个子发言 `{speaker, zh, en_keep}`。单人连续段就只有 1 个 seg。

2. **说话人名字：不确定就别编（铁律，踩过坑）**。
   - **只有当字幕本身带说话人标签、或你非常确定时**，才在 seg 里填 `speaker` 名字（渲染成加粗 `名字：` 前缀）。
   - **大多数情况：SRT 只有 `>>` 换人标记、没有名字**——「谁说的」纯属推断，容易张冠李戴。这种情况 `speaker` **一律留空 `""`**，不要硬安名字。build_doc 会在每个「换到的」子段前渲染一个中性 **🗣️** 前缀，让读者看出「换人了」，但不武断断言是谁。
   - 判据：`>>` 后面直接是台词而非 `Name:`，就是无名字字幕 → 走留空 + 🗣️ 方案。

3. **en_keep（值得展示的英文原文）**：`en_keep` 只在**值得对照英文原文的重点 seg** 上填（金句、关键论断、易被误译的原话）；普通闲聊/过渡句留 `""`。正文以中文为主线，只在有 en_keep 的段落下方补一条灰字引用块，避免逐句中英对照把篇幅翻倍。

4. **key（是否重点）**：`true` 给真正给出答案 / 抛出核心观点的实质性回合；`false` 给铺垫、提问、寒暄的回合。
   `key` 只用于挑选做节 summary 的素材，**正文不据此整段加粗**（整段加粗=噪音）；重点强调由节标题下的 💡「重点」卡片承担。
5. **每节 summary**：给每个 `section` 写一句话「重点」概述（不是搬原句，是提炼），渲染成节标题下的 💡 卡片。**只写该节自己的主旨**——不要把一句其实出现在别处的金句塞进不属于它的节。

## 精修规律（沉淀自逐轮用户 comment，务必照做）
1. **中文段落用中文全角标点**：中文子句里的 `,;:?!` 一律转全角。判定用「CJK 相邻」——只在前一个或后一个字符是汉字时才转，**保住英文 token 内的半角**（如 `productpass.com`、`SSO:`、代码）。两英文词之间但明显作中文分隔的逗号（如「Ambrosino,OpenAI」）属漏网，手工补全角。
2. **删广告/赞助口播**：插播赞助（如 productpass、WorkOS、Mercury）整段删。启发式：凡「访问 xx 网站了解更多 / 用码 xx 打折」= 广告。结尾自我推广也裁掉，只留真诚的道别/致谢。
3. **别硬翻俚语/行话，保留英文原词**：如「吃自己狗粮」硬翻别扭，写成 `dogfooding（内部试用）`；技术黑话同理，保原文加简短括注。
4. **合并被 ASR 拆断的同一句**：ASR 常在停顿处把一句连续发言拦腰切成几段。**同一说话人**、语义连续的碎段合并成一个 turn（首段吸收后段，后段清空 `{'speaker':'','zh':'','en_keep':''}`）；但**真正的短插话/附和/回声**（别人的一句「对」「没错」）保留成独立 turn，别误并。
5. **高光卡时间戳指向金句真实出现处**：`highlights[].sec` 落在这句话被真正说出的秒数，不能挂到它没出现的小节。

## 版式 v3（沉淀自真实用户反馈 + 对标 codex 更清爽的产出，重要）
这些是用户反复评论后定下的规则，务必遵守：
1. **标题保持朴素**——不要在标题里塞"中英对照""逐字稿版本"之类的强调词。
2. **正文用普通段落，不用卡片**（v3 关键改版）——旧版每 turn 一张双列 callout 卡片，用户反馈太满像卡片墙。v3 正文就是普通 `<p>`，像文档；卡片只留给节「重点」、顶部高光、头部信息框。
3. **中文为主线，英文原文作引用块**——每个 turn 先出中文普通段落，时间戳链接（+ ⭐ 若含高光）内联在首个中文段落开头；有 `en_keep` 的段落，英文原文渲染成中文下方的灰字 `<blockquote>` 引用块。
4. **换说话人标记而非硬安名字**——换到的子段前加前缀：**有确定名字**标 `名字：`，**没名字**（`>>` 无标签字幕）就用中性 **🗣️** 前缀，绝不编造说话人。
5. **绝不整段加粗**——正文任何时候不整段加粗（整段加粗=噪音）；加粗只留给说话人名字前缀、节「重点」标签。
6. **靠卡片颜色区分强调角色**——节「重点」= **light-green**（💡）；顶部高光 = **light-yellow**（⭐）；头部信息框 = **light-blue**（🎙️）。三色三角色，一眼分清。
7. **每节一张 💡 绿色「重点」卡片**——节标题下先给一句话主旨 + 可点击时间戳。
8. **高光时刻卡片**放文档顶部（黄色），每条含 label + 一句提炼 + 可跳转到金句真实秒数的时间戳；正文里对应高光段只在头部加 **⭐**。
9. 时间戳**可点击跳转**到视频对应秒；`_no_ts_sections` 里的短节（如 Cold Open 蒙太奇）隐藏全部时间戳。

版式规则已固化进 `scripts/build_doc.py`，正常情况下直接用脚本产出即可，不用手写 XML。
