# 第 6 步（必做）：生成可视化总结白板

发布逐字稿后**必须**再产出一张**提炼重点的可视化总结白板**，嵌进同一篇文档。这是标准交付物，不是可选项。

**分工铁律**：本 skill 负责**想清楚画什么**（内容结构 + 风格 + 构图），`beautiful-feishu-whiteboard` skill 负责**把它画出来**（SVG 编码、渲染、写入画板、回查线上真图）。**不要**跳过规划直接甩一堆要点给绘图 skill——那样出来的图往往是「PPT 罗列」而非「关系地图」，且风格/构图全凭绘图 skill 猜。也**不要**反过来跑去改 `beautiful-feishu-whiteboard` 的内部逻辑——它是通用引擎，本 skill 只是它的调用方。

## 什么时候做
逐字稿发布后，每次都做（不再等用户开口）。

## 流程：先规划，后交付

### 6.1 规划（本 skill 独立完成，交给绘图 skill 之前必须先定下来）
在启动 `beautiful-feishu-whiteboard` 之前，先把下面三件事**明确写出来**，作为交给它的 brief：

**① 内容结构（画什么）**
- 从 `content.json` / turns 里抽出 **4–8 个主题簇**，每簇 **3–5 条要点**（提炼成短句，别搬逐字稿原句）。
- 每个主题簇标注它在视频里的**时间戳/秒数**，方便标号排序。
- 关键：提炼出簇与簇之间的**叙事关系**——是「现状 → 方法 → 结论」的递进？还是「问题 → 拆解 → 收敛到一个决定」的收束？还是并列对比？**先想清楚主线**，因为白板的骨架就是这条主线，不是把 7 个框随便摆成网格。
- 给每个簇一个**编号**（①②③…）让读者有明确阅读顺序；编号要顺着主线走，不能乱跳。

**② 风格（画成什么样）**
- 从视频调性选一个**基调**：技术/严肃→冷静克制的配色；产品/励志→暖色、有节奏。
- 定**几个强调色**（一般 2–3 个),用来分区/标记不同「幕」或不同类别,别把所有框涂成同一个色也别塞满四五种色。
- 把这个基调作为一句话偏好传给绘图 skill，让它从自己的 35 套色板里挑一套匹配的（**不要替它指定具体色板名**，把「想要什么感觉」讲清楚即可，选板是它的职责）。

**③ 构图(怎么摆)**
- 根据 ① 的叙事关系选骨架:**递进**→单主轴 + 分支;**收束**→多路汇聚到一个决定框;**并列**→分栏对比。
- **连线只画真实存在的关系**(A 依赖/产出/对应 B),别因为「两个框挨着」就连线——乱连的箭头是噪音,宁可不连。
- 想清楚哪个框是「终点/升华」,给它更强的视觉权重(整宽横幅/主色填充)。

> 把 ①②③ 三段写成一个清晰的 brief，这就是交给绘图 skill 的输入。规划到位了，绘图 skill 才能一次画对，省掉来回返工。

### 6.2 交给 beautiful-feishu-whiteboard skill 画（用 Skill 工具启动）
把 6.1 的 brief（内容结构 + 风格基调 + 构图骨架）+ 已发布逐字稿文档的 `document_id` 交给它，明确要求：
- **嵌进本篇逐字稿文档**，而不是新建一篇文档；
- **位置铁律：白板放在「⭐ 高光时刻」和「📝 完整逐字稿」之间**，不是文档尾部。做法：`docs +fetch` 拉出块结构，找到高光区结尾的那条分割线块，用 `docs +update --command block_insert_after --block-id <分割线块>` 插入 `<h1>🗺️ 核心观点可视化总结</h1><whiteboard type="blank"></whiteboard>`，拿到新块 `block_token` 再写图。（如果先前误插到尾部，用 `block_delete` 删掉旧的那块和它的标题。）
- 走完它自己的「渲染 → 检查 → 写入画板 → 回查线上真图」全流程，最终把白板落到文档里。

构图模式的具体实现、配色体系、形状规范、映射线技法、写入画板与质量自检等细节，全部由 `beautiful-feishu-whiteboard` 的 SKILL.md 负责，这里不重复。本 skill 到 6.1 的 brief 为止。

## 仍要盯住的踩坑（转达给绘图 skill / 自检时确认）

### 文字对比度：飞书白板会把文字压深色
飞书白板**渲染时会把文字强制压成深色**，所以：
- **所有文字都放在浅色底（奶油/白/浅色）上用深墨色**，保证对比度；彩色只用于边框、标题条、色块标记、徽章、连线，**绝不把文字放到饱和色块上**；
- **别信任本地 PNG 预览的文字颜色**——它和线上不一致。必须写入后回查**线上真图**再确认对比度，有问题再改。

（这条是踩坑沉淀：曾把标题文字设成浅色放在绿/粉色条上，线上渲染全变深色压深底、几乎看不清，返工才发现。）

### 写对白板 token：overwrite 过正文会让旧白板块失效（重要，踩过坑）
若这篇文档在插入白板之后又被 `docs +update --command overwrite` 整篇重写过，**原来的 `<whiteboard>` 块会被换成一个新块、拿到新的 token**，旧 token 指向的是一个已经不在文档里的孤儿画板。往孤儿 token 写图，本地能查到图、但**用户在文档里根本看不到更新**。
- **铁律：写图前一定重新 `docs +fetch` 当前文档，从正文里解析出当前生效的 `<whiteboard token="...">`，用这个 token 写**；别用手头缓存的旧 token。
- 写完回查：既查画板线上真图，也顺带确认这个 token 就是文档正文里那一个。

### 内容要和最新正文一致
白板是正文的提炼。如果正文经过多轮评论修订（删了某句金句、改了某个说法），**白板也要跟着改**——别让白板上还挂着已经从正文删掉的旧金句。规划 6.1 时以**最新的 `content.json`** 为准。
